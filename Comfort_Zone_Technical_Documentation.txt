1.1 What the Application Does
-----------------------------
Comfort Zone is a full-stack productivity web application built with Flask 
(Python) on the backend and vanilla JavaScript on the frontend. It provides:

  â€¢ Card-Based Task Management â€” Google Keep-inspired cards with nested tasks,
    supporting full CRUD, inline editing, search, filter, sort, and bulk actions.
  â€¢ Pomodoro Timer â€” A configurable work/break timer with an SVG progress ring,
    synchronized with a video player so video plays during work and pauses during
    breaks.
  â€¢ Video Player â€” Supports both HTML5 <video> (MP4) and YouTube IFrame API.
    Persists playback position across page reloads.
  â€¢ Rich-Text Notes â€” A contenteditable-based notes editor with bold, italic,
    lists, and font sizing. Notes persist via JSON API.
  â€¢ Light/Dark Theme â€” Toggle between white-based and black-based UI themes,
    with preference persisted in localStorage.

1.2 Target Use Case
-------------------
Designed for students and self-learners who want a single, distraction-free

and note-taking â€” all without requiring a database or complex deployment.

2.2 Frontend vs Backend Responsibilities
----------------------------------------
  FRONTEND (Browser):
    â€¢ Session management via localStorage (no server sessions/cookies)
    â€¢ All UI rendering and DOM manipulation
    â€¢ Pomodoro timer logic (countdown runs entirely client-side)
    â€¢ Video playback control (HTML5 + YouTube API)
    â€¢ Theme toggling
    â€¢ Draft recovery, edit state recovery
    â€¢ Search/filter/sort (client-side after initial data fetch)

  BACKEND (Flask):
    â€¢ Serves HTML templates via Jinja2 (no dynamic rendering â€” pages are static)
    â€¢ Provides RESTful JSON API endpoints for CRUD operations
    â€¢ Handles authentication (password hashing with werkzeug)
    â€¢ Persists data to JSON files with thread-safe locking
    â€¢ Input validation via centralized validators

2.3 Request/Response Lifecycle
------------------------------
  1. User navigates to a URL (e.g., /dashboard)
  2. Flask route handler returns render_template("dashboard.html")
  3. Browser loads the HTML, which includes Bootstrap CSS, app CSS, and JS files
  4. JavaScript executes immediately:
     a. theme.js applies saved theme from localStorage
     b. state.js, api.js load as utility modules (IIFE pattern)
     c. Page controller (e.g., app.js) runs its IIFE:
        - Checks session in localStorage â†’ redirects to / if missing
        - Fetches data from API (e.g., GET /api/cards?user_id=3)
        - Renders DOM dynamically from fetched data
  5. User interactions trigger fetch() calls to the API
  6. Flask processes the request, delegates to the appropriate Manager class
  7. Manager validates input, reads/writes JSON, returns standardized result
  8. Flask converts result to JSON response with appropriate HTTP status
  9. Frontend updates DOM based on response

  3. User clicks "Start Work Session":
     - Pomodoro.nextSession() â†’ increments session counter â†’ starts work
     - Video resumes

5.4 Creating a Note â†’ Saving â†’ JSON Persistence
------------------------------------------------
  1. User types title in #cz-note-title, content in #cz-note-body
  2. Clicks "ðŸ’¾ Save Note" or presses Ctrl+S
  3. notes_editor.js â†’ handleSave()
     - Reads currentTitle and currentBody
     - Validates: title must not be empty
     - Skip save if nothing changed (same title + body as original)
  4. Decision: _currentNoteId exists AND title unchanged?
     YES â†’ API.updateNote(id, userId, title, body) â†’ PUT /api/notes/<id>
     NO  â†’ API.createNote(userId, title, body) â†’ POST /api/notes
  5. Flask route â†’ note_manager.create_note() or update_note()

     - For update: finds note by ID, checks ownership, updates fields

5.5 Note Title-Based Identity Logic (CRITICAL)
-----------------------------------------------
  The save logic uses TITLE as the primary identity signal:

  Case 1: New note (no _currentNoteId set)
    â†’ Always CREATE. Server generates new UUID.
  
  Case 2: Editing existing note, SAME title
    â†’ UPDATE the existing note via its UUID.
  
  Case 3: Editing existing note, DIFFERENT title
    â†’ CREATE a new note. The old note remains unchanged.
    â†’ This means changing a title effectively forks the note.

  This is intentional: titles serve as the user-facing identity.
  The UUID is only used internally for update/delete operations.

7.2 How JSON Files Are Read/Written
------------------------------------
  All operations follow this pattern:
  
    with self.storage.lock:           # Acquire threading.Lock
        data = self.storage.read()    # Read entire JSON file
        # ... modify data dict ...
        self.storage.write(data)      # Write entire JSON file back

  The lock ensures that concurrent requests (from Flask's threaded mode)
  don't corrupt the file. However, this is a FULL FILE READ+WRITE on
  every operation â€” not a database transaction.

7.3 User-Specific Data Handling
-------------------------------
  users.json structure:
    {
      "users": [
        {
          "id": 1,
          "username": "john",
          "password": "pbkdf2:sha256:...",   // werkzeug hash
          "cards": [
            { "id": 1, "title": "...", "created_at": "...", "tasks": [...] }
          ]
        }
      ],
      "last_user_id": 1
    }

  Cards are NESTED inside user objects. To get user's cards, the system:
    1. Reads entire file
    2. Iterates users list to find matching user_id
    3. Returns that user's "cards" array

  notes.json structure:
    {
      "notes": [
        { "id": "uuid", "user_id": 3, "title": "...", "body": "...", ... }
      ],
      "last_note_id": 0   // unused (UUIDs used instead)
    }

  Notes are stored in a FLAT list. Filtered by user_id on read.

7.4 Standardized Response Pattern
----------------------------------
  Every manager method returns:
    { "success": True/False, "data": ..., "error": "...", "status": 400 }
  
  The api_response() helper in app.py converts this to Flask JSON:
    Success: jsonify({"key": result["data"]}), 200
    Failure: jsonify({"error": result["error"]}), status_code

8. STORAGE SYSTEM (JSON)
================================================================================

8.1 Data Structures
-------------------
  User:
    { id: int, username: str, password: str(hash), cards: [Card] }
  
  Card:
    { id: int, title: str, created_at: str(ISO-8601), tasks: [Task] }
  
  Task:
    { text: str, completed: bool }
  
  Note:
    { id: str(UUID), user_id: int, title: str, body: str(HTML),
      created_at: str(ISO-8601), updated_at: str(ISO-8601) }

8.2 How Updates Happen
----------------------
  FULL FILE REPLACEMENT: Every write operation reads the entire file,
  modifies the in-memory dict, then writes the entire file back.
  There is no append-only log or partial update mechanism.

8.3 Thread Safety
-----------------
  JsonStorage uses threading.Lock (mutex). Only one thread can
  read/modify/write at a time. This prevents data corruption from
  concurrent Flask request handlers.

8.4 Risks and Limitations
-------------------------
  â€¢ File overwrite: If the process crashes mid-write, the file may be
    corrupted (partial JSON). No write-ahead log or atomic rename.
  â€¢ Concurrency ceiling: Lock serializes ALL operations. Under high
    load, requests queue up waiting for the lock.
  â€¢ Memory: Entire file loaded into memory on every request.
  â€¢ No indexing: User/card/note lookup is O(n) linear scan.
  â€¢ No relationships: Cards are nested in users (denormalized).

8.5 Why JSON Instead of a Database
-----------------------------------
  â€¢ Zero dependencies: no database server to install/configure
  â€¢ Beginner-friendly: JSON is human-readable and editable
  â€¢ Portability: entire data store is a single file
  â€¢ Sufficient for small-scale single-user use cases
  â€¢ Trade-off: not suitable for production multi-user deployment

9. FUNCTION-LEVEL EXPLANATION (Key Functions)

9.1 Frontend â€” pomodoro.js
--------------------------

  function start(workMin, breakMin, sessions)
    Purpose: Initialize and start a new Pomodoro cycle.
    Params: workMin (int), breakMin (int), sessions (int)
    Logic:
      1. Parse and validate all inputs (must be â‰¥ 1)
      2. Calculate workSec = workMin * 60
      3. Create full state object with startedAt = Date.now()
      4. Save to localStorage
      5. Start setInterval(tick, 250)
      6. Emit initial "pomodoro:tick"
    Returns: true if started, false if invalid inputs.
    Side effects: Starts interval, writes localStorage.

  function tick()
    Purpose: Core timer heartbeat â€” called every 250ms.
    Logic:
      1. Get current state; bail if not running or paused
      2. Calculate elapsed = (Date.now() - startedAt) / 1000
      3. remaining = max(0, initialTime - elapsed)
      4. Update state with new remainingTime
      5. Emit "pomodoro:tick"
      6. If remaining â‰¤ 0 â†’ handleTimerEnd()
    Side effects: Writes localStorage, fires events.

  function handleTimerEnd()
    Purpose: Handle timer reaching zero.
    Logic:
      If mode === "work" â†’ emit "pomodoro:sessionEnd"
      If mode === "break" â†’ emit "pomodoro:breakEnd"
      Both cases: stop interval, set isRunning=false
    Key design: Does NOT auto-advance. The controller decides.

  function recover()
    Purpose: Restore timer state after page reload.
    Logic:
      1. Load state from localStorage
      2. If was running and not paused: recalculate remaining time
      3. If remaining â‰¤ 0 â†’ timer expired while away â†’ handleTimerEnd()
      4. If remaining > 0 â†’ restart interval from current position
      5. Emit "pomodoro:tick" so UI renders immediately

11. EDGE CASES & HANDLING
================================================================================

11.1 Fullscreen + Popup
  Problem: If video is fullscreen when a session ends, the popup modal
  would be hidden behind the fullscreen element.
  Solution: onSessionEnd() and onBreakEnd() call
  document.exitFullscreen() BEFORE showing the modal.

11.2 Infinite Breaks
  The system allows unlimited consecutive breaks. When a break ends,
  the popup offers "Take Another Break" â€” this calls startBreak() which
  does NOT increment the session counter. The user can loop indefinitely.
  Sessions only advance when the user clicks "Start Work Session".

11.3 Duplicate Note Titles
  Creating a note with an existing title creates a SECOND note with
  the same title. There is no uniqueness constraint on note titles.
  This is by design: titles are user-facing labels, not database keys.
  However, editing a note and KEEPING the same title performs an UPDATE
  on that specific note (identified by UUID internally).

11.4 Rapid Button Clicks
  â€¢ Save button: Protected by _isSaving flag (debounce guard).
    Button is disabled during save.
  â€¢ Dashboard task operations: No explicit debounce, but each operation
    does a full renderCards() which rebuilds the DOM.
  â€¢ Timer start: Pomodoro.start() can only run if timer is not already
    running (state check in the engine).

11.5 Missing Player State
  VideoPlayer methods (play, pause, seek) check for null _videoEl
  and !_ytReady before acting. All YouTube API calls are wrapped in
  try/catch to handle destroyed player instances.

11.6 Page Reload During Active Timer
  The Pomodoro engine uses real timestamps (Date.now()), not a simple
  counter. On reload, recover() recalculates elapsed time from the
  saved startedAt timestamp. This means the timer is accurate even
  if the page was closed for minutes.

11.7 localStorage Full or Unavailable
  All localStorage operations in state.js, pomodoro.js, and
  video_player.js are wrapped in try/catch. Failures are silently
  ignored â€” the app continues working but loses persistence.

13.2 How Events Are Wired

  3. Bootstrap Events: Modal show/hide, Toast hidden.bs.toast.
